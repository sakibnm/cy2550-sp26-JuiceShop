OWASP Juice Shop — XSS & SQL Injection Challenge Set#

Prerequisites: A running instance of OWASP Juice Shop (Docker, Node.js, or cloud-hosted). A modern browser with Developer Tools. Optionally, a proxy tool such as Burp Suite Community Edition or OWASP ZAP.

Challenge 1 — Find the Score Board ⭐#

Category: Security Misconfiguration
Objective: Locate the hidden Score Board page of the Juice Shop.

Scenario:
Every good hacker starts with reconnaissance. The Juice Shop tracks your progress on a hidden Score Board page, but there is no visible link to it anywhere in the navigation menu. Your first task is to find it.

Instructions:

  1. Open the Juice Shop in your browser and explore the application.
  2. Examine the page source, JavaScript files, or route definitions to discover hidden paths.
  3. Navigate to the Score Board page to confirm you’ve found it.

What to look for:

  • How does the Angular frontend define its routes?
  • Are there commented-out HTML elements or JavaScript references that reveal hidden pages?

Learning outcome:
Understand that client-side code is never truly hidden from the user. Any route, feature, or configuration shipped to the browser can be discovered through inspection.

Challenge 2 — DOM XSS ⭐#

Category: Cross-Site Scripting (XSS)
Objective: Execute a DOM-based XSS attack using the search bar.

Scenario:
The Juice Shop has a product search feature. The search term you type is reflected directly into the page’s DOM without proper sanitization. Your goal is to inject a script that executes in the browser.

Instructions:

  1. Navigate to the Juice Shop’s main page and locate the search bar.
  2. Enter the following payload as your search term: 
    <iframe src="javascript:alert(`xss`)">
  3. Press Enter and observe the result.

Questions to answer:

  • Did the payload execute? What did you see?
  • Why does an <iframe> with a javascript: source trigger script execution?
  • Open Developer Tools → Elements. Where exactly was your payload inserted into the DOM?
  • How is this different from the server reflecting your input in the HTTP response?

Learning outcome:
Understand DOM-based XSS — where the vulnerability exists entirely in client-side JavaScript that takes user input and writes it to the page without sanitization.

Challenge 3 — Bonus Payload ⭐#

Category: Cross-Site Scripting (XSS)
Objective: Embed a richer XSS payload in the same DOM XSS vulnerability.

Scenario:
In the previous challenge you proved that the search bar is injectable. Now, instead of a simple alert, you will inject an entire embedded media player. This demonstrates that XSS is not limited to pop-up boxes — an attacker can inject any arbitrary HTML and JavaScript.

Instructions:

  1. Use the search bar again, but this time enter the following payload: 
    <iframe width="100%" height="166" scrolling="no" frameborder="no" allow="autoplay" src="https://w.soundcloud.com/player/?url=https://api.soundcloud.com/tracks/771984076&color=%23ff5500&auto_play=true&hide_related=false&show_comments=true&show_user=true&show_reposts=false&show_teaser=true"></iframe>
  2. Press Enter and observe what happens on the page.

Questions to answer:

  • What appeared on the page? Did audio play?
  • If an attacker can embed a media player, what else could they embed? Think about phishing forms, keyloggers, or cryptocurrency miners.
  • How would you, as a developer, prevent this?

Learning outcome:
Recognize that XSS impact goes far beyond alert() boxes. Arbitrary HTML injection can be used to deface pages, steal credentials via fake login forms, or execute complex JavaScript payloads.

Challenge 4 — Login Admin via SQL Injection ⭐⭐#

Category: SQL Injection
Objective: Log in as the administrator without knowing the password.

Scenario:
The Juice Shop’s login form sends your email and password to the server, which constructs an SQL query to check your credentials. The query is not parameterized, meaning your input is placed directly into the SQL string. Exploit this to bypass authentication and log in as the admin.

Instructions:

  1. Navigate to the Login page (/#/login).
  2. In the Email field, enter a payload that will manipulate the SQL query. Think about how you can make the WHERE clause always evaluate to true, or how you can comment out the password check.
  3. Enter anything in the Password field.
  4. Click “Log in.”

Hints (reveal progressively):

  • Hint 1: What character is used to terminate a string in SQL?
  • Hint 2: What does -- do in SQL?
  • Hint 3: The query looks like: SELECT * FROM Users WHERE email='<input>' AND password='<hash>'

Questions to answer:

  • What payload did you use? Write out the full SQL query that the server executed with your input substituted in.
  • Why did the server return the administrator’s account specifically?
  • What is the fundamental coding mistake that made this possible?
  • What is a parameterized query (or prepared statement), and how would it prevent this attack?

Learning outcome:
Understand classic SQL Injection on a login form. Learn why string concatenation in SQL queries is dangerous and how prepared statements eliminate the vulnerability.

Challenge 5 — Reflected XSS ⭐⭐#

Category: Cross-Site Scripting (XSS)
Objective: Perform a reflected XSS attack via a URL parameter.

Scenario:
After placing an order in the Juice Shop, you receive a tracking page that displays your order’s tracking ID. The tracking ID is taken directly from a URL parameter and rendered on the page. If the value is not sanitized, an attacker could craft a malicious URL that executes JavaScript when a victim clicks it.

Instructions:

  1. Place an order in the Juice Shop (add any item to the basket, go through checkout).
  2. After the order is confirmed, look at the order confirmation/tracking page and examine the URL.
  3. Identify the URL parameter that controls the tracking ID displayed on the page.
  4. Replace the tracking ID value in the URL with: 
    <iframe src="javascript:alert(`xss`)">
  5. Load the modified URL.

Questions to answer:

  • How is reflected XSS different from the DOM XSS you performed in Challenge 2?
  • If you sent this crafted URL to another user (e.g., via email or chat), what would happen when they click it?
  • Why is this called “reflected” XSS?
  • What HTTP header could help mitigate this type of attack?

Learning outcome:
Understand reflected XSS — where malicious input is sent to the server via a URL/request and reflected back in the response. Learn why this is dangerous in phishing and social engineering scenarios.

Challenge 6 — Login Jim via SQL Injection ⭐⭐⭐#

Category: SQL Injection
Objective: Log in as Jim’s user account using SQL Injection.

Scenario:
You’ve already proven that the login form is vulnerable to SQL injection. In Challenge 4 you logged in as the first user in the database (the admin). Now your target is a specific user: Jim. You need to craft an injection that targets Jim’s account rather than just returning the first row.

Instructions:

  1. First, do some reconnaissance. Browse the Juice Shop’s product reviews or other public-facing content to discover Jim’s email address format (e.g., jim@juice-sh.op).
  2. Navigate to the Login page.
  3. Craft an SQL injection payload for the Email field that will select Jim’s account specifically and bypass the password check.
  4. Enter anything as the password and log in.

Questions to answer:

  • How did you find Jim’s email address? What does this tell you about information leakage?
  • Write out the exact payload you used and the resulting SQL query.
  • How does this differ from the ' OR 1=1-- approach? Why is a targeted injection more dangerous in a real-world scenario?

Learning outcome:
Learn to perform targeted SQL Injection against a specific user account. Understand that information disclosure (like visible email addresses) combined with injection vulnerabilities dramatically increases risk.

Challenge 7 — Login Bender via SQL Injection ⭐⭐⭐#

Category: SQL Injection
Objective: Log in as Bender’s user account using SQL Injection.

Scenario:
This is a reinforcement challenge. Repeat the targeted SQL injection technique, but this time against Bender’s account. There is one twist: unlike Jim, Bender chose a strong password, so even if you extract his password hash later, brute-forcing it would be impractical. SQL injection to bypass the password check entirely is the only practical approach.

Instructions:

  1. Determine Bender’s email address by exploring the application (product reviews, admin panel if accessible, or by educated guessing based on the email format you’ve already seen).
  2. Craft the appropriate injection payload on the login form.
  3. Verify you are logged in as Bender.

Questions to answer:

  • Why does a strong password NOT protect Bender from this attack?
  • If you were advising the Juice Shop’s developers, what single code change would fix the login form for ALL users at once?
  • What is the difference between authentication bypass and credential theft?

Learning outcome:
Reinforce targeted SQLi skills and understand that password strength is irrelevant when the authentication logic itself is broken.

Challenge 8 — Client-side XSS Protection Bypass ⭐⭐⭐#

Category: Cross-Site Scripting (XSS)
Objective: Perform a persisted XSS attack by bypassing client-side input validation.

Scenario:
The Juice Shop’s user registration form validates the email field on the client side using JavaScript. However, if no equivalent validation exists on the server, an attacker can bypass the browser’s checks entirely by intercepting and modifying the HTTP request.

Instructions:

  1. Open the Registration page (/#/register).
  2. Fill in the form with valid-looking data and a proper email address.
  3. Before submitting, open your browser’s Developer Tools (Network tab) or configure a proxy (Burp Suite / ZAP).
  4. Submit the form and intercept the outgoing request.
  5. Modify the email field in the request body to: 
    <iframe src="javascript:alert(`xss`)">
  6. Forward the modified request to the server.
  7. Navigate to the Administration page (/#/administration) while logged in as admin to see the injected payload rendered.

Questions to answer:

  • Why did the client-side validation not stop you?
  • Where does the XSS payload get stored? Why is stored/persisted XSS more dangerous than reflected XSS?
  • What is the security principle demonstrated here? (Hint: “Never trust the client.”)
  • How should the server validate this input?

Learning outcome:
Understand that client-side validation is a user experience feature, not a security control. Learn the critical importance of server-side input validation and output encoding.

Challenge 9 — API-only Persisted XSS ⭐⭐⭐#

Category: Cross-Site Scripting (XSS)
Objective: Inject a persisted XSS payload by directly calling the REST API, without using the frontend at all.

Scenario:
The Juice Shop’s REST API allows product data to be modified. The API endpoint does not properly validate or sanitize input, and the frontend displays product descriptions without encoding. By sending a crafted API request, you can inject JavaScript into a product description that will execute for every user who views that product.

Instructions:

  1. Using Developer Tools, Burp Suite, Postman, or curl, identify the API endpoint for products (e.g., GET /api/Products/).
  2. Choose a product (note its ID).
  3. Send a PUT request to /api/Products/<ID> with the following JSON body: 
    {
  "description": "<iframe src=\"javascript:alert(`xss`)\">"
}
    Make sure Content-Type is set to application/json.
  4. Visit the Juice Shop’s main page and locate the modified product to see the payload execute.

Questions to answer:

  • The frontend forms might sanitize input, but does the API? What does this tell you about where security controls should live?
  • This is a stored/persisted XSS. How many users could be affected compared to a reflected XSS?
  • Why should APIs enforce the same (or stricter) validation rules as frontend forms?
  • What is output encoding, and how would it prevent the injected HTML from executing?

Learning outcome:
Learn that APIs are an independent attack surface. Security controls must be enforced at the API/server level, not just in the frontend UI. Understand the severity of stored XSS vulnerabilities.

Challenge 10 — Exfiltrate the Database Schema ⭐⭐⭐#

Category: SQL Injection
Objective: Use SQL Injection on the search endpoint to extract the full database schema.

Scenario:
The product search feature at /rest/products/search?q= is vulnerable to SQL injection. Unlike the login form (which returns a yes/no authentication result), this endpoint returns data in the HTTP response. This opens the door to UNION SELECT attacks, where you append a second query that pulls data from entirely different tables.

Instructions:

  1. Open your browser or proxy and navigate to the search API directly: /rest/products/search?q=test
  2. Try injecting a single quote ' into the q parameter. Observe the error message — it will reveal the SQL query structure and the database engine (SQLite).
  3. Determine the correct way to break out of the query. (Hint: the query uses double parentheses.)
  4. Craft a UNION SELECT attack to extract schema definitions from sqlite_master. You will need to match the number of columns in the original query (there are 9).
  5. Your final payload should look something like: 
    ')) UNION SELECT sql,2,3,4,5,6,7,8,9 FROM sqlite_master--

Questions to answer:

  • What information did the error messages reveal? Why are verbose error messages a security risk?
  • What is sqlite_master and why does it exist?
  • What tables and columns did you discover in the schema? Which ones look like high-value targets?
  • What is a UNION SELECT attack, and why does the number of columns need to match?
  • How would you prevent this? (Think: parameterized queries, WAF rules, error handling.)

Learning outcome:
Learn UNION-based SQL Injection — a technique for extracting arbitrary data from a database. Understand how error messages assist attackers and why information disclosure through error handling must be controlled.

Challenge 11 — Retrieve All User Credentials ⭐⭐⭐⭐#

Category: SQL Injection
Objective: Use SQL Injection to dump all user email addresses and password hashes from the database.

Scenario:
This is the capstone challenge. In Challenge 10 you extracted the database schema and discovered the Users table along with its column names. Now you will use that knowledge to craft a UNION SELECT that retrieves every user’s email and password hash. This simulates a full credential theft — one of the most damaging outcomes of a SQL Injection vulnerability.

Instructions:

  1. Using the same vulnerable search endpoint (/rest/products/search?q=), craft a UNION SELECT that pulls id, email, and password from the Users table.
  2. You will need to position these columns correctly so they appear in readable fields of the product search results. Columns 2 and 3 of the product results are displayed as text on the page.
  3. Prefix your injection with a non-existent product name to suppress normal search results and only show your injected data: 
    nonexistent')) UNION SELECT 1,email,password,4,5,6,7,8,9 FROM Users--
  4. Examine the results. You should see a list of all users with their password hashes.

Questions to answer:

  • How many user accounts did you retrieve? List a few of the email addresses you found.
  • What hashing algorithm do the passwords appear to use? (Hint: look at the hash length and format.)
  • Pick one of the hashes and try looking it up in an online hash database (e.g., CrackStation). Were any passwords weak enough to be reversed?
  • In a real-world breach, what would an attacker do with this data?
  • What are the multiple layers of defense that could have prevented this entire attack chain? (Think: input validation, parameterized queries, password hashing algorithms, least-privilege database accounts, network segmentation.)

Learning outcome:
Experience the full impact of SQL Injection — from initial discovery through schema extraction to complete credential theft. Understand defense-in-depth: even if injection is possible, strong password hashing (bcrypt/argon2), proper database permissions, and monitoring can limit the damage.
